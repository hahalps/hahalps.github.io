消息处理模块

消息处理模块的作用是根据当前的状态生成信息并发送信息，并具有处理信息的能力。这个模块的话可以看作是两大部分组成，一个是创建信息，另一个的话是处理信息。

熟悉的环节，函数功能的介绍。（写这个的原因一是理清模块的整体是什么，另一个的方便以后借鉴里面的实现，同时的话也是想自己的心慢下来，不要的太过于浮躁，慢慢品味这个“作品”吧）

1，int int_to_char(int i, unsigned char c[4])

作用是把一个32位的整型1转换成char[]存储的方式，作者是用减法，除法再加上模运算实现的的，写起来又长又不好看，自己就改写成位运算的形式。

2，int char_to_int(unsigned char c[4])

作用是char[]转换成一个32位的整数返回。

3，int creat_handshake_msg(char *info_hash, char *peer_id, Peer *peer)

这个函数的作用把握手信息写进peer的输出缓冲区当中。先看握手信息的格式为:<info_hash><peer_id>。

pstrlen的值固定为19，占用一个字节；pstr固定为“BitTorrent protocol”,占用19个字节；reserved是用来扩充bt协议的，占用8的字节；最后的两个的话是info_hash以及peer_id, 分别占用20个字节，所以总的占用字节数为1+19+8+20+20=68个字节。

创建消息的共同点在于会先检测输出缓冲去的剩余大小是否大于等于消息的长度，如果是的话，才把信息写进输出缓冲区当中去。接着的话就是把信息写进缓冲区当中了，最后更新一下缓冲区的长度。

4， creat_keap_alive_msg(Peer *peer)

keep_alive的信息长度固定，为4个字节，消息的格式为<len=0000>。这个函数是让与peer没有数据交互的情况还保持连接，因为协议本身会检测，如果在两分钟之内和peer没有交互的话，程序会关闭与这个peer的连接。实现的很简单，在输出缓冲区有剩余的情况下把缓冲区的4位清空为0。

5，int creat_choke_interested_msg(int type, Peer *peer)

根据type的内容创建choke信息, unchocke信息, interested信息以及not interested消息当中的一个，四个所对应的值分别为0，1，2，3。他们四者有共同的信息格式<len=0001>。

6，int creat_have_msg(int index, Peer *peer)

其实写到这里创建消息的方式都是一样的，先检测针对peer的输出缓冲区的剩余空间是否充足，然后把消息按照消息的格式写进缓冲区就可以了。这里的话先写下消息的格式吧：<len==0005><id=4>, 消息的长度为4+1+4=9个字节。

7，int creat_bitfield_msg(char *bitfield, int bitfield_len, Peer *peer)

哦，反应过来，len的数值跟它后面的内容的长度是有关的。消息的格式为：<len=0001+x><id=5>, x是位图信息的字节数。这里的话我们可以算下可以传递的最大字节数是998, 算下的话998*8=7984个piece的信息，而每个piece的大小我们假设为256kb的话，交换的最大信息量为1996Mb。哦买噶，错了错了，这里len的四位表示的是一个32位的整数啊，最大值不是9999,啊，还算错了，不改了，当作是给自己的一个提醒吧！

8， int creat_request_msg(int index, int begin, int length, Peer *peer)

创建request信息，请求peer第index个piece的，从第begin开始的长度为length的slice。格式的话是<len=0013><id=6>。

9，int creat_piece_msg(int index, int begin, int *block, int b_len, Peer *peer)

是针对于request信息的回复信息。格式的话就是<len=0009+x><id=7>。这里的x对应的是block的长度。

10，int creat_cancle_msg(int index, int begin, int length, Peer, *peer)

撤销掉之前的request信息，跟request的信息的区别在于两者之间的编号吧！格式的话是<len=0013><id=8>

11, int creat_port_msg(int port, Peer *peer)

消息的格式为：<len=0003><id=9>, 这里的话监听的端口号占用两个字节，也就是16位bit。而这个消息的一般不需要理会。

12， int is_complete_message(unsigned char *buff, unsigned int len, int *ok_len)

判断缓冲区有多少条完成的信息，通过ok_len返回最后一条完整信息的结束位置。程序的话先判断定长的信息，在判断不定长的信息，定长的信息的话结合常量表的技巧来判断，不定长的信息的话通过\加上\的信息来判断就可以了。

13，int process_handshake_msg(Peer *peer, unsigned char *buff, int len)

这个函数是用来处理握手信息的。处理的方式先检测info_hash值是否一样，如果不一样的话说明下载的不是同一个东西，并终止连接；如果一样的话，先记录peer_id,再根据当前的状态是否发送信息，如果处于初始化状态的话，向对方发送一个握手信息，如果处于半握手状态的话，只需把当前状态修改为握手状态就行了，相当于之前我已经发送过一次信息，而这次的话只是对方收到信息的回复。最后的话更新一下最近收到peer的信息时间start_timestamp。

14，int process_keep_alive_msg(Peer *peer, unsigned char *buff, int len)

处理keep_alive信息。处理的方式是直接更新start_timestamp就行了。

15，int process_choke_msg(Peer *peer, unsigned char *buff, int len)

这个函数是处理收到来自peer的choke信息。处理的方式是如果当前与peer存在连接且之前的状态peer_choking为0状态，这就需要先把peer_chocking的状态置为1，且因为被peer给阻塞了，所以有关从peer下载的变量都需要置为0，这其中包括有最后一次下载的时间last_down_timestamp, 下载的字节数down_count以及下载的速率down_rate都置为0。最后的话同样更新一下start_timestamp.

16，int process_unchoke_msg(Peer *peer, unsigned char *buff, int len)

这个函数的话处理收到来自于peer的unchoke信息。如果本身处于未阻塞的状态的话，只需更新一下start_timestamp的时间就可以了；如果peer_choking是处于阻塞的状态且双方是处于连接的状态的话，更新下peer_choking信息，并判断是否对peer感兴趣，如果感兴趣的话向peer发送request信息就行了，最后把跟下载相关的变量置为0，因为计数从现在开始咯。

17，int process_insterested_msg(Peer *peer, unsigned char *buff, int len)

处理收到的interested信息。当处于数据交换的状态，通过位图信息发现peer对客户端感兴趣且客户端没有把peer阻塞的情况下，向该peer发送一条unchoke信息。大概的样貌是这样子的，（打个比喻），就是有人喜欢你，他向你发送一条表达爱意的信息之后，而你是根据什么做出决定的呢？是当你处于想谈恋爱的阶段且你对对方也有好感的情况下，你才接收这条信息，且给他回复条信息表明我们之间可以建立起“数据交换”的状态了。

18， int process_uninterested_msg(Peer *peer, unsigned char *buff, int len)

处理收到的uninterseted信息。收到这个信息说明peer对客户端不感兴趣了，而我们需要做的事就是更新状态之外，还需要撤销来自该peer的请求列表。最后更新一下时间戳。

19， int process_have_msg(Peer *peer, unsigned char *buff, int len)

处理收到的have信息。在双方处于数据交换的情况下，先根据收到的have信息更新一下该peer的位图信息，其次的话判断是否对该peer感兴趣，如果是的话向该peer发送interested信息。其实随后的过程就是peer收到信息发现没有将其阻塞的话，就发送unchoke信息，当收到unchoke信息且自身对peer感兴趣，就可以正式向该peer发送request信息了。

20， int process_cancel_msg(Peer *peer, unsigned char *buff, int len)

这个函数对应的操作就是在链表当中找到某一结点进行删除操作。

21, int process_bitfield_msg(Peer *peer, unsigned char *buff, int len)

处理收到的位图信息。主要的操作是提取信息当中的位图信息并把位图信息放到peer结构体的bitfield[]中，最后的更新当前流程处于什么样的状态，如果是处于握手状态的话，则发送信息给对方；如果已经处于位图发送状态的话，则进入数据交换的状态。最后，更新一下时间戳。

22， int process_request_msg(Peer *peer, unsigned char *buff, int len)

处理收到的请求信息。做法类似一道这样的题，给你一条链表以及一个结点，先判断该结点是否在链表当中，如果不在的话，把结点接到链表的末尾。更新时间。

23，int process_piece_msg(Peer *peer, unsigned char *buff, int len)

处理收到的piece信息。先检测是否发送过请求，如果有的话，再把数据写进缓冲去当中去，再更新一下跟下载有关的属性。最后为了实现片段选择算法，再发送一个请求给peer。

24， int parse_response(Peer *peer)

这个的话处理peer发送给客户端的消息。具体的做法就是根据消息的类型调用上面提到的处理函数。注意的一点是处理完所有的信息之后记得把缓冲区清空。

25，int parse_response_uncomplete_msg(Peer *peer, int ok_len)

这个是处理不完整信息的。先把完整的部分通过上面的函数进行处理，再把不完整的部分copy到数组开头并更新下缓冲区的长度。

26，int prepare_send_have_msg()

告知所有的peer，老子已经下载好的信息，要的话过来拿就行了，不必跑那么远去其他地方拿。这里的have_piece_index[]不知道在哪里声明的。

27, int creat_response_msg(Peer *peer)

主动向peer发送信息。这个函数可以分为三大部分，第一部分，当双方处于刚建立tcp连接的时候，向该peer发送握手信息；第二部分，当处于握手状态的话，向该peer发送位图信息；第三部分那就是处于数据交换的状态咯，处于这种状态的话，当客户端没有将peer阻塞且被请求列表不为空的时候，就向该pee上传数据并更新跟上传有关的变量， 最后的处理是状态图的最后一个状态CLOSING, 如果超过三分钟没有接收到任何信息的就关闭连接，且如果45秒没有发送或者接收消息的话，则发送keep_alive信息，只是用来保持，可能没有进行数据的上传或者是下载。

28，void discard_send_buff(Peer *peer)

用到linger结构体。用来断开socket的连接。{1， 0}含义是close()不会发送未发送的数据，强制退出。
